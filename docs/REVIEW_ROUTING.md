# Review Routing

Use this guide to decide which maintainers or domain reviewers should be asked
to review a pull request, issue, maturity promotion, or safety question.

The repository starts with placeholder CODEOWNERS because GitHub usernames and
teams are not known yet. Once the public repository owner is set, update
`.github/CODEOWNERS` with real owners and keep this document in sync.

Run `python3 tools/review_candidates.py --limit 12` when seeding the first
`maturity-review` queue. The report ranks seed skills by local static evidence
and maps them to the domain areas below.

## Path Routing

| Path | Primary Review | Notes |
| --- | --- | --- |
| `skills/` | Registry maintainer plus domain reviewer when the skill is specialized. | Check manifest quality, examples, risk label, public-safe assumptions, and references. |
| `collections/` | Registry maintainer. | Confirm every referenced skill exists and the adoption path is coherent. |
| `site-adapters/` | Registry maintainer plus adopter or site-adapter reviewer. | Verify that all values are public-safe and no private hostnames, accounts, projects, or policy details are included. |
| `schemas/` | Registry maintainer plus tooling reviewer. | Treat schema changes as compatibility-sensitive. |
| `tools/` and `src/` | Tooling reviewer plus registry maintainer. | Confirm CLI, validation, generated registry output, and tests still agree. |
| `.github/` | Maintainer with release or repository-admin context. | Check CI, labels, issue templates, rulesets, and branch protection expectations. |
| `docs/` | Documentation maintainer or relevant domain reviewer. | Route governance, release, and maturity docs to maintainers; route domain guides to domain reviewers. |
| `registry/` and generated docs | Registry maintainer. | Prefer changes generated by the documented tooling rather than hand edits. |

## Domain Routing

Ask for domain review when a change requires operational judgment beyond the
registry format. Use [Domain Reviewers](DOMAIN_REVIEWERS.md) as the public
recruitment and ownership map for these areas:

| Domain | Examples |
| --- | --- |
| Scheduler and allocation | Slurm submission, arrays, accounting, time limits, partitions, reservations, preemption, and requeue behavior. |
| Storage and data movement | Quotas, scratch, archival staging, rsync, Globus, object storage, checksum workflows. |
| Software environments | Modules, compiler/MPI stacks, licensed software jobs, Conda, Python virtualenvs, Spack, EasyBuild. |
| Containers | Apptainer/Singularity build and runtime patterns, image portability, bind paths. |
| Workflow engines | Nextflow, Snakemake, nf-core, checkpoint/restart, staging plans. |
| MPI, GPU, and performance | MPI launch, MPI fabric diagnostics, GPU binding diagnostics, rank binding diagnostics, hybrid MPI/OpenMP layouts, OpenMP affinity, CUDA/ROCm visibility, NCCL, profiling, AI training launchers. |
| Scientific domains | Bioinformatics, molecular dynamics, CFD, weather, or other domain-specific workflows. |
| Facility operations | Node health, usage reporting, module tree health, training cluster preparation, support triage. |

## Risk Routing

- Low-risk changes can be reviewed by a registry maintainer.
- Medium-risk changes need review for resource impact, file output, cleanup,
  portability, and site assumptions.
- Facility operations, admin, security, destructive cleanup, accounting, quota,
  node-state, or shared-system behavior needs safety review before merge or
  maturity promotion.
- Any new high-risk skill should start as an RFC or safety review issue before
  implementation.

## Label Routing

Use labels to make the review queue visible:

| Label | Route |
| --- | --- |
| `skill-request` | Registry maintainer, then domain reviewer if the request is accepted. |
| `site-adapter` | Registry maintainer plus adopter or site-adapter reviewer. |
| `safety-review` | Maintainer with production HPC operations or security context. |
| `maturity-review` | Domain reviewer with evidence from an actual review, pilot, or public run. |
| `adoption` | Maintainer plus the adopting site or research group contact. |
| `schema-change` | Registry and tooling maintainers. |
| `integration` | Tooling reviewer plus the downstream tool or workflow maintainer. |
| `release` | Release maintainer. |
| `governance` | Maintainers responsible for policy, review, and community process. |
| `needs-triage` | Registry maintainer should confirm scope, sensitivity, and next action. |
| `needs-domain-review` | Domain reviewer should check portability, safety, assumptions, and references. |
| `good-first-issue` | Registry maintainer should confirm scope is small and public-safe. |

## Pull Request Steps

1. Match the changed paths to the path routing table.
2. Check skill domain, maturity, and risk labels.
3. Add the narrowest reviewer set that covers format, domain, and safety.
4. Link any safety review, maturity review, RFC, adoption report, or public
   evidence in the pull request.
5. Confirm `make check` passes before merge.

## CODEOWNERS Update Checklist

After the repository owner and maintainer teams exist:

- Replace placeholder examples in `.github/CODEOWNERS` with real GitHub users
  or teams.
- Add broad fallback ownership for the repository.
- Add path-specific owners for `schemas/`, `tools/`, `.github/`, `skills/`,
  `collections/`, and `site-adapters/`.
- Add domain-specific entries only when a stable team is responsible for that
  area.
- Keep branch protection and CODEOWNERS settings aligned. If code-owner review
  is required, make sure the listed teams are active and able to review.
